## 0. 最重要的总原则

不要一开始就把所有问题都交给微调。

正确判断顺序：

```text
Prompt 能解决 -> 先用 Prompt
需要查资料 -> 用 RAG
需要实时数据 -> 用 API / 工具
需要稳定语气、格式、流程、判断标准 -> 用 LoRA / QLoRA
```

客服机器人推荐组合：

```text
Qwen Instruct 基座模型
+ System Prompt
+ RAG 知识库
+ API / 工具调用
+ QLoRA 微调
+ 后端 HTTP API
+ 前端网页调用
```

一句话分工：

```text
Prompt 管“规则边界”
LoRA 管“怎么答”
RAG 管“答什么资料”
API 管“实时状态”
```

## 1. 大模型微调的概念，包括 SFT

### 1.1 什么是微调

微调就是在一个已经训练好的大模型基础上，继续用你的业务数据训练它，让它更稳定地完成某类任务。

它不是从零训练一个模型，而是在已有基座模型上做定制。

```text
基座模型: 已经学过大量通用语言能力
微调数据: 你的业务样本
微调目标: 让模型学会固定任务模式、输出格式、语气和判断标准
```

### 1.2 微调适合解决什么

适合：

```text
固定输出格式
固定客服语气
固定任务模式
固定判断标准
稳定 JSON 输出
意图识别
转人工判断
拒答边界
降低 prompt 长度
提升某类任务稳定性
```

不适合：

```text
实时订单状态
实时库存
实时价格
频繁变化的售后政策
大量产品文档硬背
公司知识库问答
```

这些应该交给：

```text
RAG: 产品文档、FAQ、售后政策、公司制度
API: 订单、物流、库存、退款进度
Prompt: 身份、规则、边界
```

### 1.3 什么是 SFT

SFT 是 `Supervised Fine-Tuning`，也就是监督式微调。

意思是：你给模型很多“标准输入 + 标准答案”，让模型学习应该如何回答。

客服场景里，SFT 样本可以长这样：

```json
{
  "messages": [
    {
      "role": "system",
      "content": "你是某品牌客服，回答简洁、礼貌、准确。不知道就引导用户转人工或查询系统。"
    },
    {
      "role": "user",
      "content": "我买的商品可以退货吗？"
    },
    {
      "role": "assistant",
      "content": "可以的。若商品符合退货条件，可申请退货。请提供订单号，我可以继续帮您确认。"
    }
  ]
}
```

如果你要做结构化客服助手，也可以输出 JSON：

```json
{
  "messages": [
    {
      "role": "system",
      "content": "你是客服消息分析助手，只输出合法 JSON。"
    },
    {
      "role": "user",
      "content": "我买的手机壳收到是破的，怎么处理？"
    },
    {
      "role": "assistant",
      "content": "{\"intent\":\"damaged_item\",\"urgency\":\"medium\",\"sentiment\":\"negative\",\"need_human\":false,\"reply\":\"很抱歉收到商品有损坏。请提供订单号和商品照片，我们会为您安排补发或退款。\"}"
    }
  ]
}
```

### 1.4 微调 vs 从头训练

| 方式 | 适合场景 | 数据量 | 算力 | 你的阶段是否推荐 |
|---|---|---:|---:|---|
| Prompt | 规则简单、快速验证 | 不需要训练数据 | 无 GPU | 推荐先试 |
| RAG | 知识库问答、政策文档、产品说明 | 文档为主 | 低 | 推荐 |
| LoRA / QLoRA | 固定语气、格式、判断标准 | 100 到 10K 条 | 单卡可做 | 推荐 |
| 全量 SFT | 深度领域定制 | 更高 | 多卡 | 暂不推荐 |
| 从头训练 | 自定义模型架构或超大专有语料 | 极大 | 集群 | 不推荐入门 |

## 2. LoRA、QLoRA 与量化

### 2.1 LoRA 是什么

LoRA 是一种轻量微调方法。

它不重新训练整个大模型，而是冻结原模型，只训练一小组新增的低秩矩阵。

核心公式：

```text
W' = W + alpha/r * B*A
```

含义：

```text
W: 原模型权重，冻结不训练
A/B: 新增的小矩阵，参与训练
r: LoRA rank，秩
alpha: 缩放系数
```

直观理解：

```text
原模型 = 大脑
LoRA adapter = 给大脑贴上的小补丁
```

LoRA 训练产物通常是：

```text
adapter_config.json
adapter_model.safetensors
```

注意：LoRA adapter 不能独立运行，必须搭配原始基座模型。

运行方式：

```text
方式 1:
base model + LoRA adapter

方式 2:
base model + LoRA adapter -> merge 成完整模型
```

### 2.2 LoRA 模型能不能单独运行

通常不能。

更准确地说：

```text
LoRA adapter 本身不能单独运行。
它只是基座模型上的增量权重，不是完整模型。
```

LoRA adapter 目录里通常只有：

```text
adapter_config.json
adapter_model.safetensors
```

这类文件只记录“在原模型哪些层上加了哪些小矩阵”，不包含完整大模型权重，所以不能像完整模型一样直接推理。

### 2.3 怎么让 LoRA 可以运行

有两种方式。

方式 1：运行时同时加载基座模型和 LoRA adapter。

```text
Qwen3-8B-Instruct
+ customer-service-lora-adapter
=> 可以推理
```

这种方式适合：

```text
训练后测试
多套 adapter 切换
同一个 base model 服务多个业务风格
不想生成一份完整大模型副本
```

方式 2：把 LoRA adapter 合并回基座模型，导出完整模型。

```text
Qwen3-8B-Instruct
+ customer-service-lora-adapter
-> merge
-> Qwen3-8B-CustomerService-Finetuned
```

这种方式适合：

```text
正式部署
转 GGUF / AWQ / GPTQ
给 Ollama / llama.cpp / vLLM 使用
减少运行时 adapter 加载复杂度
```

目录判断：

```text
只有 adapter_config.json + adapter_model.safetensors
=> LoRA adapter，不能单独运行

有 config.json + model-00001-of-000xx.safetensors + tokenizer
=> 通常是完整模型，可以单独运行
```

一句话：

```text
LoRA adapter 不能单独运行；
要么 base + adapter 一起加载；
要么 merge 成完整模型后再单独运行。
```

### 2.4 QLoRA 是什么

QLoRA = 量化加载基座模型 + LoRA 训练。

更准确地说：

```text
LoRA:
基础模型按 fp16 / bf16 加载，冻结
训练 LoRA adapter

QLoRA:
基础模型按 4bit 加载，冻结
训练 LoRA adapter
```

所以：

```text
QLoRA 是训练方式
LoRA adapter 是训练结果
```

不要理解成：

```text
先导出 LoRA，再量化成 QLoRA
```

正确理解是：

```text
用 QLoRA 方式训练 LoRA adapter
训练完成后仍然通常导出 LoRA adapter
部署时再决定是否 merge 和推理量化
```

### 2.5 量化是什么

量化是把模型参数从高精度数字压缩成低精度数字，用更少显存保存和运行模型。

常见精度：

| 精度 | 每个参数大约占用 | 常见用途 |
|---|---:|---|
| FP32 | 32 bit / 4 bytes | 传统训练，高精度 |
| FP16 | 16 bit / 2 bytes | 推理、训练 |
| BF16 | 16 bit / 2 bytes | 训练首选，动态范围更好 |
| INT8 | 8 bit / 1 byte | 推理压缩 |
| INT4 / NF4 | 4 bit / 0.5 byte | QLoRA、低显存推理 |

QLoRA 常用的是：

```text
4bit NF4
```

NF4 不是普通 INT4，它更适合神经网络权重的分布。

### 2.6 训练量化和部署量化的区别

训练阶段：

```text
bitsandbytes 4bit NF4
用于 QLoRA 训练
```

部署阶段：

```text
AWQ / GPTQ: GPU 推理量化
GGUF: llama.cpp / Ollama / CPU 或混合推理
INT8: 更稳但压缩率较低
```

简单选择：

```text
训练用 NF4
部署用 AWQ / GPTQ
本地轻量使用用 GGUF
```

### 2.7 LoRA 和 QLoRA 怎么选

核心依据：GPU 显存。

```text
显存充足 -> 优先 LoRA
显存紧张 -> 用 QLoRA
```

粗略参考：

| 显存 | 建议 |
|---:|---|
| 8GB | QLoRA，小模型优先 |
| 12GB | QLoRA 7B |
| 16GB | QLoRA 7B / 14B 较现实 |
| 24GB | LoRA 7B，QLoRA 14B |
| 48GB | LoRA 14B，探索更大模型 |
| 80GB | LoRA 32B 或更大模型探索 |

对你的客服微调：

```text
第一优先: 模型别太大，数据质量高
第二优先: 能稳定跑完训练
第三优先: 再考虑 LoRA vs QLoRA
```

## 3. 大模型选择、RAG 区别与结合

### 3.1 Dense vs MoE 怎么选

核心取舍：

```text
Dense 更稳、更简单
MoE 更省推理成本、更适合大规模服务，但工程复杂度更高
```

Dense 模型：

```text
每次推理激活全部参数
微调简单
推理延迟稳定
LoRA / QLoRA 支持成熟
适合入门客服微调
```

MoE 模型：

```text
总参数很大，但每次只激活部分专家
适合高并发、多领域复杂任务
部署和微调更复杂
路由稳定性需要注意
```

你的阶段建议：

```text
先 Dense
后 MoE
```

推荐路线：

```text
学习阶段: Qwen3-8B-Instruct + QLoRA
正式阶段: Qwen3-14B-Instruct + QLoRA
进阶阶段: Qwen3-30B-A3B + QLoRA
暂不建议: 一开始上 24B / 27B / 32B dense
```

更细的 Dense / MoE 选型：

| 场景 | 优先选择 | 原因 |
|---|---|---|
| 第一次做客服微调 | Dense 7B / 8B | 稳定、资料多、容易排错 |
| 4090 24GB 上做正式客服版 | Dense 14B + QLoRA | 能力更强，显存还能控制 |
| 高并发、多业务线客服 | MoE | 激活参数少，推理成本更有优势 |
| 数据量小、任务单一 | Dense | MoE 路由不一定能学稳定 |
| 要快速验证 LoRA 效果 | Dense | LoRA 兼容性和工具链最成熟 |
| 已有成熟部署和评测体系 | 可以考虑 MoE | 需要能处理路由、吞吐、量化和服务稳定性 |

实用判断：

```text
小团队 / 入门 / 单一客服场景:
Dense 优先

大规模服务 / 多领域任务 / 高并发成本敏感:
MoE 再考虑
```

### 3.2 Base 和 Instruct 怎么选

客服机器人建议直接选 Instruct。

```text
Base:
更像续写模型
适合继续预训练、研究、深度定制

Instruct:
已经对齐过指令和聊天格式
更适合问答、客服、助手、多轮对话
```

客服微调建议：

```text
Qwen3-8B-Instruct
Qwen3-14B-Instruct
```

### 3.3 Qwen、Gemma 和其他模型怎么选

当前客服微调主线建议仍然是 Qwen。

| 模型系列 | 适合场景 | 优点 | 注意点 |
|---|---|---|---|
| Qwen | 中文客服、中文结构化输出、本地微调 | 中文强、生态成熟、Qwen chat template 清晰 | 注意训练和推理模板一致 |
| Gemma | 英文/多语言实验、轻量模型练手 | 模型小、适合 Unsloth + QLoRA 教学实验 | 中文客服不是首选 |
| Llama | 英文生态、通用微调研究 | 社区资料多、部署工具成熟 | 中文业务通常要更多评测 |
| Mistral / Mixtral | 高性能英文任务、MoE 实验 | 推理效率和能力不错 | 中文客服要谨慎验证 |

Gemma 可以作为学习 QLoRA 的练手模型：

```text
Gemma 2B / 7B
+ Unsloth
+ QLoRA
+ SFTTrainer
```

但如果目标是中文客服机器人，优先级建议：

```text
第一优先: Qwen Instruct
第二优先: Llama / Mistral 做横向对比
第三优先: Gemma 做训练框架练习
```

Gemma 路线适合你理解训练代码和框架，不一定是最终产品模型。

### 3.4 LoRA 为什么不能跨 7B / 24B 通用

LoRA 是贴在某个具体基座模型结构上的补丁，所以强绑定基座模型。

不能跨尺寸通用的核心原因：

```text
层数不同:
7B 和 24B Transformer 层数不同，LoRA 挂载位置数量对不上。

hidden size 不同:
模型内部向量宽度不同，比如 4096、5120、6144。
LoRA 矩阵尺寸依赖 hidden size，尺寸不同就加载不了。

attention 结构不同:
比如 q_proj / k_proj / v_proj / o_proj，GQA / MQA / head 数量不同。
结构不同，LoRA 权重形状不同。

MLP 结构不同:
比如 gate_proj / up_proj / down_proj。
不同模型中间维度不同，也会导致 LoRA 不兼容。

参数命名不同:
LoRA 会记录挂在哪些层名上，例如 model.layers.0.self_attn.q_proj。
模型命名不同就找不到位置。

tokenizer / 词表不同:
不同模型切词方式不同。即使 LoRA 勉强能加载，输入分布变了，效果也会掉。

chat template 不同:
多轮对话转成模型输入的模板不同，训练和推理不一致会导致效果异常。
```

可以复用的情况：

```text
同一个模型
同一个尺寸
同一个结构
只是 FP16 / 4bit / 8bit 量化版本不同
```

不建议复用的情况：

```text
Qwen3-8B LoRA -> Qwen3-14B
Qwen3-8B LoRA -> Qwen3-24B
Qwen3-8B LoRA -> Llama / Mistral
Base 训练的 LoRA -> Instruct 随便套
```

长期真正要保存的是：

```text
训练数据
system prompt
chat template
训练参数
评测集
业务规则
```

LoRA 权重不是最可迁移的资产，数据和配置才是。

### 3.5 LoRA 和 RAG 的核心区别

最核心区别：

```text
LoRA 是改模型的“行为习惯”
RAG 是给模型“查资料”
```

场景区别：

| 技术 | 解决什么 | 客服例子 |
|---|---|---|
| LoRA / QLoRA | 怎么判断、怎么表达、怎么输出 | 意图识别、固定 JSON、品牌语气、转人工判断 |
| RAG | 回答基于哪些资料 | 售后政策、产品说明、FAQ、活动规则 |
| API / 工具 | 实时状态是什么 | 订单状态、物流、库存、退款进度 |
| Prompt | 当前对话规则 | 身份、边界、禁止编造、输出约束 |

不是：

```text
LoRA = 只改语气
RAG = 让模型真正学会新知识
```

而是：

```text
LoRA = 训练行为模式
RAG = 检索知识上下文
```

### 3.6 客服机器人最佳组合

第一阶段：只做 QLoRA。

```text
目标: 跑通训练流程
内容: 固定语气、固定 JSON、意图识别、拒答边界
不做: 复杂 RAG、不接真实订单系统
```

第二阶段：加入 RAG。

```text
目标: 接入产品文档、FAQ、售后政策
内容: 用户问规则时，先查知识库，再回答
```

第三阶段：加入 API。

```text
目标: 查询真实业务状态
内容: 订单、物流、退款、库存、账号信息
```

最终结构：

```text
用户
  -> 前端网页
  -> 后端 API
  -> 意图识别
  -> RAG 检索
  -> 业务 API 查询
  -> Qwen + LoRA adapter 生成回复
  -> 返回前端
```

## 4. 技术框架选择、步骤与算力选择

### 4.1 技术框架怎么选

第一阶段推荐：

```text
LLaMA-Factory
```

原因：

```text
支持 Qwen
支持 LoRA / QLoRA
支持 SFT
支持 Web UI / 命令行
支持 adapter 导出
支持 merge 合并模型
不需要从零写训练代码
```

第二阶段可选：

```text
Unsloth
```

适合：

```text
更快训练
更省显存
尝试更大模型
消费级 GPU 优化
```

进阶再学：

```text
TRL + PEFT
Axolotl
DeepSpeed
vLLM
```

一句话理解：

```text
TRL: 负责 SFT / DPO 等训练流程
PEFT: 负责 LoRA / QLoRA 等高效微调方法
```

### 4.2 训练和部署工具分工

| 环节 | 推荐工具 |
|---|---|
| 微调入门 | LLaMA-Factory |
| 快速省显存训练 | Unsloth |
| 自己写训练代码 | TRL + PEFT |
| 大规模训练 | DeepSpeed / FSDP |
| 高吞吐服务 | vLLM |
| 本地轻量运行 | Ollama / llama.cpp |
| GPU 推理量化 | AWQ / GPTQ |
| CPU / 本地格式 | GGUF |

注意：

```text
训练时不要从 Ollama 模型开始。
微调训练建议下载 Hugging Face / ModelScope 的 Transformers 格式模型。
```

训练需要这些文件：

```text
config.json
tokenizer 文件
model-00001-of-000xx.safetensors
generation_config.json
```

Ollama 更偏推理，不是微调训练的起点。

### 4.3 YOLO 视觉模型路线

YOLO 和大语言模型微调不是一类任务，但它是理解“从预训练模型做任务微调”的好对照。

```text
LLM 微调:
输入文本 / 对话
输出文本 / JSON / 工具调用意图
常用 SFT / LoRA / QLoRA

YOLO 微调:
输入图片
输出检测框 / 类别
常用 Ultralytics YOLO 训练
```

YOLO 适合：

```text
目标检测
缺陷检测
商品识别
人车检测
工业视觉
图片中定位某类物体
```

YOLO 不适合：

```text
客服问答
文档问答
文本分类
多轮对话
知识库问答
```

YOLOv8 模型选型参考：

| 模型 | 参数量 | 特点 | 适合场景 |
|---|---:|---|---|
| YOLOv8n | 3.2M | 最小最快 | 嵌入式、树莓派、实时低算力 |
| YOLOv8s | 11.2M | 轻量 | 边缘设备 |
| YOLOv8m | 25.9M | 平衡 | 一般项目首选 |
| YOLOv8l | 43.7M | 更高精度 | 服务器部署 |

## 5. 大模型微调的完整步骤

### 5.1 第一版目标

第一版不要追求最终效果。

目标是跑通闭环：

```text
模型能下载
数据能读取
训练能开始
显存不爆
adapter 能导出
推理能加载
评测能对比
流程能复现
```

### 5.2 推荐执行路线

```text
1. 搭环境
   Python / Conda / Git / PyTorch CUDA / LLaMA-Factory

2. 下载模型
   Qwen3-8B-Instruct
   建议使用 Hugging Face / ModelScope 的 Transformers 格式

3. 准备数据
   先做 50-100 条客服 SFT 样本
   统一 system prompt
   统一 chat template

4. 小样本试跑
   确认能加载模型
   确认不爆显存
   确认能导出 adapter

5. 正式训练
   扩到 300-1000 条高质量数据
   使用 QLoRA 4bit NF4

6. 训练后测试
   base model + LoRA adapter
   用固定评测集测试

7. 迭代
   格式不稳 -> 补格式样本
   语气不稳 -> 补话术样本
   意图错判 -> 补边界样本
   事实错误 -> 优先改 RAG，不要让 LoRA 背动态知识

8. 部署
   后端加载模型
   接 RAG / API
   包 HTTP 接口
   前端网页调用
```

### 5.3 数据量建议

```text
50-100 条:
跑通流程

300-1000 条:
第一版可用客服微调

1K-10K 条:
更完整客服 SFT

50K+:
更大规模指令微调，不是第一阶段重点
```

原则：

```text
高质量 1000 条 > 低质量 10000 条
宁可少而精，不要多而脏
```

客服数据应覆盖：

```text
售前咨询
退货退款
物流订单
投诉安抚
转人工
拒答边界
固定 JSON 输出
多轮追问
敏感/越权问题
不知道时不编造
```

### 5.4 推荐数据格式

建议使用 messages / ShareGPT / ChatML 类格式：

```json
{
  "messages": [
    {
      "role": "system",
      "content": "你是某品牌客服，回答简洁、礼貌、准确。遇到订单、物流、退款状态时，不要编造结果，应要求用户提供订单号或调用系统查询。"
    },
    {
      "role": "user",
      "content": "我的快递三天没动了，是不是丢了？"
    },
    {
      "role": "assistant",
      "content": "{\"intent\":\"logistics_delay\",\"urgency\":\"medium\",\"sentiment\":\"negative\",\"need_human\":false,\"reply\":\"抱歉让您久等了。请提供订单号或物流单号，我可以帮您查询最新物流状态；如果物流长时间无更新，我们会协助您联系快递核实。\"}"
    }
  ]
}
```

### 5.5 起步训练参数

适合 RTX 4090 24GB 的 Qwen3-8B / 14B 客服 QLoRA 起步参数：

```yaml
base_model: Qwen3-8B-Instruct
method: QLoRA
stage: sft
load_in_4bit: true
quantization: NF4
compute_dtype: bf16
chat_template: qwen
max_seq_length: 2048

lora_rank: 16
lora_alpha: 32
lora_dropout: 0.05
target_modules:
  - q_proj
  - k_proj
  - v_proj
  - o_proj
  - gate_proj
  - up_proj
  - down_proj

learning_rate: 0.0002
epochs: 2-3
per_device_train_batch_size: 1
gradient_accumulation_steps: 8
gradient_checkpointing: true
```

参数解释：

| 参数 | 建议 | 解释 |
|---|---|---|
| `base_model` | `Qwen3-8B-Instruct` 起步 | 基座模型，LoRA 强绑定它 |
| `method` | `QLoRA` | 4bit 加载基座，训练 adapter |
| `load_in_4bit` | `true` | 开启 4bit 量化加载 |
| `quantization` | `NF4` | QLoRA 常用量化格式 |
| `compute_dtype` | `bf16` | 4090 支持，训练更稳 |
| `lora_rank` | `16` 起步，`32` 增强 | LoRA 容量，越大越能学，但更吃显存 |
| `lora_alpha` | `2 * rank` | 常见配置，rank 16 对应 alpha 32 |
| `lora_dropout` | `0.05`，小数据可 `0.1` | 防过拟合 |
| `target_modules` | attention + MLP | attention 必加，MLP 可提升表达能力 |
| `learning_rate` | `1e-4` 到 `2e-4` | LoRA / QLoRA 常见学习率 |
| `epochs` | `2-3` | 客服小数据先别训太久 |
| `max_seq_length` | `2048` 起步 | 长对话再升到 4096 |
| `batch_size` | `1` 起步 | 4090 上先稳 |
| `gradient_accumulation_steps` | `8` | 模拟更大 batch |
| `gradient_checkpointing` | `true` | 省显存 |

### 5.6 什么时候调参数

第一轮不要急着调参数，先看错误类型。

```text
JSON 格式不稳:
补更多 JSON 样本，统一输出格式。

语气不统一:
补品牌话术样本，统一 system prompt。

意图错判:
补相似边界样本，例如退款 vs 换货、投诉 vs 咨询。

回答编造政策:
不要靠 LoRA 背政策，改用 RAG。

过拟合:
减少 epoch，加 dropout，增加多样化样本。

loss 不下降:
检查数据格式、学习率、chat template、标签是否干净。

显存爆:
减小 batch_size，开启 gradient_checkpointing，使用 QLoRA，缩短 max_seq_length。
```

### 5.7 训练后怎么用

训练完成通常得到：

```text
LoRA adapter
```

目录判断：

```text
只有 adapter_config.json + adapter_model.safetensors
=> LoRA adapter，不能独立运行

有 config.json + model-00001-of-000xx.safetensors + tokenizer
=> 通常是完整模型，可以独立运行
```

部署选择：

```text
测试阶段:
base model + adapter 分开加载

正式阶段:
base model + adapter merge 成完整模型

轻量部署:
merge 后转 GGUF / AWQ / GPTQ

服务部署:
vLLM / Transformers / llama.cpp / Ollama
```

## 8. 你的推荐路线总结

最终建议：

```text
模型:
Qwen3-8B-Instruct 起步
Qwen3-14B-Instruct 正式阶段
Gemma 可作为 QLoRA / Unsloth 练手模型
MoE 等流程成熟后再考虑

训练:
QLoRA
4bit NF4
LoRA rank 16 起步
alpha 32
dropout 0.05
max_seq_length 2048
batch_size 1
gradient_accumulation_steps 8

框架:
LLaMA-Factory 起步
Unsloth 加速
TRL + PEFT 进阶理解原理
YOLO 视觉检测用 Ultralytics，不和客服 LLM 微调混在一起

数据:
先 50-100 条跑通
再 300-1000 条做第一版
后续扩到 1K-10K 条

产品架构:
LoRA 负责客服行为模式
RAG 负责知识库
API 负责实时状态
Prompt 负责规则边界
```

一句话结论：

```text
先用 Qwen3-8B-Instruct + LLaMA-Factory + QLoRA 跑通客服 SFT；
再把数据、评测、RAG、API 补齐；
最后再考虑 14B、MoE、Gemma 横向对比、vLLM 和部署量化。
```

视觉方向单独走：

```text
如果任务是图片目标检测，用 YOLOv8m + Ultralytics 起步；
如果任务是客服文本问答，用 Qwen + QLoRA + RAG 起步。
```
