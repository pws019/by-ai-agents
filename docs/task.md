# 数据集扩充计划（data）—— QLoRA 训练数据

## 0. 范围澄清（2026-07-16 重新确定）

本期目标不是训练一个"全能客服大模型"，而是训练一个 **严格贴合生产 system prompt 边界**
的 QLoRA：决策 + 话术，不训练具体业务知识、不训练虚构的工具返回结果。

生产环境实际使用的 system prompt（来自项目截图，作为本期唯一边界依据）：

```text
1. 你是专业、耐心、克制的电商客服。
2. 如果用户询问物流、快递、签收、派送、催件，并提供了订单号，必须先调用 logisticsLookupTool 查询。
3. 如果用户询问物流但没有提供订单号，先请用户提供订单号或收件手机号后四位。
4. 如果用户询问商品说明、使用方法、保养方式、售后政策、退换货流程、质保等非实时知识，优先调用
   knowledgeRagTool 检索知识库，再基于 relevantContext 和 sources 回答。
5. knowledgeRagTool 只用于非实时知识，不要用于实时订单、物流、库存、价格、退款进度等查询。
6. 不要编造订单、物流、库存、价格、退款进度或售后政策；知识库没有命中时，要说明暂未查到准确
   依据，并建议转人工或补充商品/订单信息。
7. 回答时先承接用户情绪，再说明查询结果或需要的信息，最后给出下一步处理方式。
8. 如果用户询问与电商客服无关的问题，例如吃喝玩乐、闲聊、编程、学习、新闻等，要友好但明确地
   说明自己只能协助订单、物流、商品说明、售后等客服相关事项，并邀请用户提供相关问题。
```

## 0.1 v1 训练后的实测发现（2026-07-16，重大调整依据）

v1（23 场景，A-E 五大类，46 mock + 460 sft）训练完成后做了两件事：

1. Mock 集预测：ROUGE-1 77.48 / ROUGE-2 60.02 / ROUGE-L 73.10 / BLEU-4 55.16，B/D/E 类表现符合预期。
2. **本机用真实 system prompt 原文 + 真实 tools schema（`logisticsLookupTool`）实测 A 类场景**：
   给模型"已提供订单号的物流问题"+ 完整 tools 定义，基座+LoRA 的输出仍然是纯自然语言
   "您好，我现在为您查询该订单当前的物流状态"，**完全没有吐出 `<tool_call>`**，说明 LoRA
   训练的自然语言应付习惯，实际盖过/干扰了基座模型本该有的原生工具调用能力。

结论：项目采用的是 **agentic 架构**（模型自己根据 system prompt 里列出的 tools 决定调用
哪个工具，不是后端代码先分类再决定），因此：

- **A 类（物流·已提供订单号，对应 `logisticsLookupTool`）和 C 类（商品说明/使用方法/保养
  方式/售后政策/退换货流程/质保等知识类问题，对应 `knowledgeRagTool`）本质上都是"该由模型
  自主决策调用哪个工具"的场景。这轮教模型用自然语言去应付，属于教错了行为，且会干扰原生工具
  调用能力，弊大于利。**
- 在没有确定真实工具参数 schema、无法训练规范 `sharegpt+tools` 格式的函数调用样本之前，
  A、C 两类 **整体不训练**，完全交给基座模型的原生 agentic 工具调用能力处理，不做任何自然
  语言层面的替代训练。
- B 类（物流·未提供订单号）安全：没有订单号，任何机制下都调不了 `logisticsLookupTool`，
  这是工具调用的前置判断，不存在"跟原生能力打架"的问题，继续训练。
- D 类（库存/价格/退款进度）安全：system prompt 明确写了这些没有对应工具（`knowledgeRagTool`
  不能用于这些），没有工具在跟它抢位置，继续训练。
- E 类（离题拒答）无害但收益有限：system prompt 第 8 条已经把这个行为讲得很清楚，基座模型
  大概率靠指令遵循就能做好，训练主要是为了话术统一和边缘案例泛化，不是必须，本轮保留。
- 之前生成的 mock 测试集（含 A/C 类内容）已被删除，需要针对新范围（B/D/E）全部重新生成。

**后续 Phase 4（待办，不在本轮）**：等 tool-service 定好真实参数 schema 后，针对 A、C 两类
补一批 `sharegpt` + `tools` 格式的真实函数调用样本（含成功/失败两种分支），到时候再决定是否
训练模型自主选择工具这件事本身，或者继续完全信任基座原生能力。

## 0.2 输出风格新原则（2026-07-16 确认）——模板化、收敛话术

之前 v1 数据刻意让同一场景内的话术措辞多样化（避免"换个订单号就是一条新数据"）。这次反过来：

**同一场景内的 output，要收敛成同一个模板骨架，只有触发信息（订单号/具体缺失字段/具体话题词）
不同，句式结构、用词、语气基本一致，减少大模型自由发挥的空间。**

原因：客服机器人需要给用户一致、可预期的体验，同一类问题措辞发散会显得不专业、不可控。

每个场景在下面的场景清单里都附一个"标准模板"，生成数据时以这个模板为骨架去填充变化点，
而不是像 v1 那样每条都重新组织句子结构。

## 1. 场景清单（v2，仅 B/D/E 三大类）

每个场景：4 条 mock + 40 条 sft（相比 v1 的 2+20 翻倍，弥补砍掉 A/C 两类后的数据量，
并加强剩余场景的触发句式覆盖面）。

### B. 物流类 · 未提供订单号 → 引导提供订单号或手机号后四位

标准模板：
```text
您好，{共情句，呼应用户的情绪/诉求}，请提供订单号或收件手机号后四位，我会{用它查询/尽快为您核实}最新的物流状态{视情况加：并协助跟进催促}。
```

- [x] B1. 直接问物流/快递，完全没给任何标识
- [x] B2. 问物流但只给了模糊信息（收货人姓名、大概时间等，非订单号/手机号）
- [x] B3. 催件但拒绝/忘记提供订单号
- [x] B4. 情绪激动、反复追问但仍未提供订单号

### D. 无工具覆盖的实时数据类 → 不编造，引导转人工或补充信息

标准模板：
```text
您好，{共情句}，{库存/价格/退款进度}属于实时信息，我这边暂时无法直接给出准确{数量/金额/进度}，建议为您转接人工客服进一步核实，请问是否需要为您申请转接？
```

- [x] D1. 库存查询
- [x] D2. 价格/优惠咨询
- [x] D3. 退款进度查询

### E. 离题无关问题 → 礼貌拒答，说明范围，邀请提出客服相关问题

标准模板：
```text
您好，{简短承接/感谢}，不过{具体话题}不在我能协助的范围内，我主要负责订单、物流、商品说明、售后等客服相关事项。请问有什么购物或订单方面的问题需要帮忙吗？
```

- [x] E1. 吃喝玩乐/生活闲聊
- [x] E2. 编程/技术类问题
- [x] E3. 学习/知识科普类问题
- [x] E4. 新闻/时事类问题
- [x] E5. 其他杂项无关请求（占卜、翻译、代写作业等）

共 12 个场景 → mock 48 条 + sft 480 条。
