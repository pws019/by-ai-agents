# 数据集扩充计划（data）—— QLoRA 训练数据

## 0. 范围澄清（2026-07-16 重新确定）

本期目标不是训练一个"全能客服大模型"，而是训练一个 **严格贴合生产 system prompt 边界**
的 QLoRA：决策 + 话术，不训练具体业务知识、不训练虚构的工具返回结果。

生产环境实际使用的 system prompt（来自项目截图，作为本期唯一边界依据）：

```text
1. 你是专业、耐心、克制的电商客服。
2. 如果用户询问物流、快递、签收、派送、催件，并提供了订单号，必须先调用 logisticsLookupTool 查询。
3. 如果用户询问物流但没有提供订单号，先请用户提供订单号或收件手机号后四位。
4. 如果用户询问商品说明、使用方法、保养方式、售后政策、退换货流程、质保等非实时知识，优先调用
   knowledgeRagTool 检索知识库，再基于 relevantContext 和 sources 回答。
5. knowledgeRagTool 只用于非实时知识，不要用于实时订单、物流、库存、价格、退款进度等查询。
6. 不要编造订单、物流、库存、价格、退款进度或售后政策；知识库没有命中时，要说明暂未查到准确
   依据，并建议转人工或补充商品/订单信息。
7. 回答时先承接用户情绪，再说明查询结果或需要的信息，最后给出下一步处理方式。
8. 如果用户询问与电商客服无关的问题，例如吃喝玩乐、闲聊、编程、学习、新闻等，要友好但明确地
   说明自己只能协助订单、物流、商品说明、售后等客服相关事项，并邀请用户提供相关问题。
```

关键决定：
- **不虚构工具返回结果**：output 只训练"该不该调用工具 / 该问什么 / 该怎么措辞"，不在 output
  里编造 relevantContext、物流节点原文等虚构内容。
- **output 采用纯自然语言方案**（2026-07-16 确认）：类 A（已给订单号）的 output 只写自然语言
  确认，例如"好的，已收到订单号，我现在为您查询物流信息，请稍候"，不写结构化 `<tool_call>`
  JSON。原因：工具名称（logisticsLookupTool / knowledgeRagTool）已在 system prompt 里定死，
  但参数 schema（如 order_id 还是 orderId）尚未确定，现在编结构化格式风险是后续对不上真实
  schema、需要重做。本轮数据教的是不随 schema 变化的部分：意图识别、何时追问订单号、语气、
  "先接情绪→给信息/需求→给下一步"的结构。
- **后续 Phase 4（待办，不在本轮）**：等 tool-service 定好真实参数 schema 后，补一批
  `sharegpt` + `tools` 格式的真实函数调用样本，和本轮数据一起或增量训练，教模型吐出机器可解析
  的 `<tool_call>{"name":...,"arguments":{...}}</tool_call>`。本轮数据不会因此浪费。
- **知识库类问题本期一律转人工**：knowledgeRagTool 尚未接入，第 4 条里"商品说明/使用方法/
  保养方式/售后政策/退换货流程/质保"这些问题，本期训练目标是"识别为知识类问题 → 引导转人工"，
  不是"识别为知识类问题 → 离题拒答"，也不是"编知识回答"。
- 之前设计的隐私套取、骚扰辱骂、性骚扰、威胁投诉、诱导编造、刷单作弊、钓鱼链接、医疗安全、
  企业采购、发票等 48 个场景超出本期 system prompt 边界，本期不做，数据也已随环境回滚清空，
  不需要额外清理。

## 1. 场景清单（本期唯一版本）

每个场景：2 条 mock + 20 条 sft（2026-07-16 确认放大样本量）。

### A. 物流类 · 已提供订单号 → 触发 logisticsLookupTool
