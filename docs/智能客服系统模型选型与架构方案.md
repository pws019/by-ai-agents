# 智能客服系统模型选型与架构方案

## 1. 当前结论

你的本机适合做：

```text
本地开发
本地推理验证
RAG / API / 前后端联调
成品模型 + LoRA adapter 的效果测试
量化模型部署体验验证
```

你的本机不适合作为主要 QLoRA 训练机器。QLoRA 训练建议租用 NVIDIA GPU 服务器完成，本机负责下载、运行、评测和产品开发。

推荐主线：

```text
本机推理基座: Qwen3-8B
训练方式: 租 GPU 做 QLoRA / LoRA
客服架构: System Prompt + RAG + API 工具调用 + QLoRA + 后端 HTTP API + 前端网页
第一版目标: 先把 Qwen3-8B 客服版跑通，不急着上 14B / 30B
```

## 2. 你的电脑能带动什么模型

已确认机器配置：

```text
设备: MacBook Pro
芯片: Apple M5 Pro
内存: 48GB 统一内存
GPU: Apple M5 Pro 16 核 GPU
CPU: 15 核
```

Apple Silicon 的优势是统一内存容量较大，适合跑量化后的本地大模型。限制是它不能使用 CUDA / bitsandbytes 的主流训练栈，所以不适合作为正式 QLoRA 训练环境。

### 2.1 本机推理推荐表

| 模型规模 | 本机建议 | 推荐精度 | 适合用途 |
|---|---|---|---|
| 1.5B / 3B / 4B | 很轻松 | Q4 / Q5 / Q8 | 快速原型、低延迟客服、前后端联调 |
| 7B / 8B | 推荐主力 | Q4 / Q5 / Q6 | 中文客服第一版、RAG 联调、LoRA 效果验证 |
| 14B | 可跑，但不建议第一版主力 | Q4 / Q5 | 更强效果评测、低并发本地体验 |
| 30B MoE | 可以探索，不建议产品第一版 | Q4 | 高级对比实验，需要更细的部署调优 |
| 32B Dense 及以上 | 不建议本机作为主力 | Q4 | 能否运行取决于上下文长度和推理框架，速度通常不理想 |

最稳的选择：

```text
第一优先: Qwen3-8B
第二优先: Qwen3-4B-Instruct-2507 / Qwen3-4B 级别模型
第三优先: Qwen3-14B 做对比评测
暂不建议: 一开始上 30B / 32B / MoE 做主产品
```

### 2.2 为什么推荐 Qwen3-8B

Qwen3-8B 对你的目标比较平衡：

```text
中文能力好
客服场景够用
本机 48GB 内存能比较稳地跑量化版本
QLoRA 训练成本可控
RAG + 工具调用后，不需要模型死记所有业务知识
```

注意模型命名：

```text
Qwen3-8B: 文本主线的后训练模型，可直接用于聊天/指令任务
Qwen3-8B-Base: 基础模型，适合继续训练或研究
Qwen3-VL-8B-Instruct: 视觉语言模型，不是纯文本客服主力模型
```

客服系统不要优先选 `Qwen3-VL-8B-Instruct`。VL 模型适合看图、OCR、截图理解、多模态问答；纯文本客服、RAG、工具调用优先用文本 LLM。

## 3. 本机部署形态

本机建议使用量化推理，不建议直接跑 FP16。

推荐优先级：

```text
Ollama: 最方便，适合快速聊天和本地 API 验证
LM Studio: 适合图形化测试模型效果
llama.cpp / GGUF: 适合深入控制量化和部署
MLX / mlx-lm: 适合 Apple Silicon 原生推理和轻量实验
```

本机建议模型格式：

```text
GGUF Q4_K_M: 体积小、速度好、适合日常验证
GGUF Q5_K_M: 效果更稳，速度和内存仍可接受
GGUF Q6_K: 质量更好，但资源占用更高
MLX 4bit: Apple Silicon 上体验通常较好
```

第一版建议：

```text
本机先跑 Qwen3-8B 的 Q4 或 Q5 量化版本
服务端先封装 OpenAI-compatible HTTP API
前端只调用你自己的后端，不直接调用模型服务
```

## 4. QLoRA 训练服务器建议

QLoRA 训练放到租用 GPU 服务器。

### 4.1 Qwen3-8B 训练

推荐配置：

| GPU | 是否推荐 | 说明 |
|---|---|---|
| RTX 4090 24GB | 推荐 | 8B QLoRA 很合适，性价比高 |
| L40S 48GB | 推荐 | 更稳，可拉长上下文和 batch |
| A100 40GB / 80GB | 推荐 | 稳定，适合正式训练 |
| H100 80GB | 过强但很好 | 成本高，训练快 |

Qwen3-8B QLoRA 起步参数：

```yaml
model: Qwen3-8B
method: QLoRA
quantization: 4bit NF4
compute_dtype: bf16
lora_rank: 16
lora_alpha: 32
lora_dropout: 0.05
learning_rate: 1e-4
max_seq_length: 2048
batch_size_per_device: 1-2
gradient_accumulation_steps: 8-16
epochs: 2-3
```

### 4.2 14B 训练

14B 可以作为第二阶段，不建议第一阶段就上。

推荐配置：

```text
最低: 4090 24GB + QLoRA + 小 batch + gradient checkpointing
更稳: L40S 48GB / A100 40GB
正式: A100 80GB / H100 80GB
```

14B 的价值是回答质量和复杂指令能力可能更好，但训练、部署、评测成本都会上升。客服系统第一版通常先用 8B 把数据、RAG、工具调用和评测闭环跑通。

## 5. 架构是否合理

你提出的架构是合理的：

```text
Qwen Instruct 基座模型
+ System Prompt
+ RAG 知识库
+ API / 工具调用
+ QLoRA 微调
+ 后端 HTTP API
+ 前端网页调用
```

更准确的职责分工：

| 模块 | 负责什么 | 不负责什么 |
|---|---|---|
| System Prompt | 身份、边界、回答原则、工具使用规则 | 不背大量知识 |
| RAG 知识库 | FAQ、政策、产品文档、售后规则 | 不处理实时订单状态 |
| API / 工具调用 | 订单、物流、退款、库存、会员信息 | 不生成自然语言风格 |
| QLoRA | 客服语气、固定流程、意图判断、结构化输出 | 不记频繁变化的业务知识 |
| 后端 HTTP API | 鉴权、会话、RAG 编排、工具编排、模型调用 | 不把密钥暴露给前端 |
| 前端网页 | 聊天 UI、登录态、消息展示、人工客服入口 | 不直接访问模型密钥 |

一句话：

```text
Prompt 管规则
RAG 管资料
API 管实时状态
QLoRA 管客服行为习惯
后端管编排
前端管交互
```

## 6. 推荐系统架构

```text
用户浏览器
  -> 前端聊天页面
  -> 后端 HTTP API
      -> 鉴权 / 限流 / 会话管理
      -> 意图识别
      -> RAG 检索
      -> 工具调用
          -> 订单系统
          -> 物流系统
          -> 退款系统
          -> 库存系统
      -> LLM 推理服务
          -> Qwen3-8B
          -> customer-service LoRA adapter 或 merge 后模型
      -> 回复后处理
  -> 前端展示
```

后端建议拆成几个清晰模块：

```text
chat-api: 对外聊天接口
conversation-store: 会话历史
rag-service: 文档切分、向量检索、重排
tool-service: 订单/物流/退款等业务 API 封装
model-gateway: 统一模型调用入口
eval-service: 离线评测和回归测试
```

## 7. 微调数据应该训练什么

QLoRA 数据不要拿来硬背知识库。

适合训练：

```text
品牌客服语气
多轮对话流程
用户意图分类
是否需要转人工
是否需要调用工具
固定 JSON 输出格式
拒答和安全边界
投诉安抚话术
缺少信息时如何追问
```

不适合训练：

```text
实时价格
实时库存
物流状态
订单状态
经常变化的活动规则
大段产品说明
售后政策全文
```

这些应该放到：

```text
实时状态 -> API / 工具调用
稳定文档 -> RAG
临时规则 -> System Prompt 或配置中心
```

## 8. 第一版落地路线

### Phase 1: 本机推理和 API 验证

目标：

```text
本机跑通 Qwen3-8B 量化模型
后端提供 /chat 接口
前端能发消息并展示回复
```

建议输出：

```text
模型服务: Ollama / llama.cpp / MLX 任选其一
后端: OpenAI-compatible client 调用模型
前端: 简单客服聊天页面
```

### Phase 2: RAG 知识库

目标：

```text
导入 FAQ / 售后政策 / 产品文档
实现切分、embedding、向量检索
把检索结果作为上下文传给模型
```

建议先用小数据集验证：

```text
20-50 条 FAQ
3-5 篇政策文档
10-20 个回归测试问题
```

### Phase 3: 工具调用

目标：

```text
让模型能判断什么时候查订单、查物流、查退款
后端实际调用 mock API
模型基于工具结果回复用户
```

先 mock，再接真实系统。

### Phase 4: QLoRA 训练

目标：

```text
租 GPU 训练 Qwen3-8B 客服 LoRA adapter
训练客服语气、流程、结构化输出和工具调用倾向
```

数据规模建议：

```text
Smoke test: 50-100 条
第一版: 300-1000 条
可上线评估版: 1000-5000 条
成熟版: 5000-10000 条，并配套评测集
```

### Phase 5: 合并、量化、部署

目标：

```text
base model + LoRA adapter 评测
必要时 merge 成完整模型
导出 GGUF / AWQ / GPTQ
本机和服务器分别做部署验证
```

本机部署优先 GGUF / MLX。

服务器部署优先：

```text
vLLM: NVIDIA GPU 服务部署
TGI: Hugging Face 生态部署
SGLang: 高性能推理和 agent 场景
```

## 9. 模型选择建议

### 9.1 第一版

```text
模型: Qwen3-8B
训练: QLoRA
部署: 本机 Q4/Q5 量化验证，服务器 vLLM 或 Ollama/llama.cpp 视场景选择
```

这是最推荐路线。

### 9.2 轻量版

```text
模型: Qwen3-4B-Instruct-2507 或同级 4B 模型
训练: QLoRA
部署: 本机和低成本服务器都更轻松
```

适合低成本、低延迟、简单客服。

### 9.3 增强版

```text
模型: Qwen3-14B
训练: QLoRA
部署: 更依赖服务器 GPU，本机只做量化低并发验证
```

适合第一版跑通后做质量提升对比。

### 9.4 暂缓路线

```text
Qwen3-VL-8B-Instruct: 多模态客服才需要
Qwen3-30B-A3B: 后续实验，不作为第一版主线
32B Dense 以上: 本机不是理想主力部署环境
```

## 10. 最终推荐方案

最终建议：

```text
1. 本机用 Qwen3-8B 量化模型做客服系统开发和推理验证
2. 后端先实现统一 /chat API，前端只调用自己的后端
3. 第一阶段先做 System Prompt + 基础会话
4. 第二阶段加入 RAG，让模型回答基于知识库
5. 第三阶段加入工具调用，处理订单、物流、退款等实时状态
6. 第四阶段租 4090 24GB / L40S 48GB / A100 做 QLoRA
7. 训练产物先以 LoRA adapter 方式评测，再决定是否 merge 和量化
8. 第一版不要上 VL，不要上 30B，不要让 LoRA 背业务知识
```

一句话版本：

```text
用 Qwen3-8B 做主力成品模型，本机负责跑和验，租 GPU 负责 QLoRA 训练；客服知识交给 RAG，实时状态交给 API，客服风格和流程交给 LoRA。
```
